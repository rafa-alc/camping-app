It enforces scope discipline, clean architecture, validation, and a clear delivery summary. ---  # MVP delivery workflow for CampIn  ## Purpose  Use this skill when implementing a feature, refactor, or iteration inside the CampIn MVP repository.  This skill exists to prevent common agent mistakes: - inventing features outside scope - mixing business logic into UI - adding unnecessary complexity - breaking the MVP architecture - returning code without proper validation or explanation  Always follow this workflow together with the repository `AGENTS.md`.  ## Step 1 — Re-read project constraints  Before coding, restate internally the critical constraints:  - solo-developer project - MVP V1 only - no backend - no auth - no social features - no AI features - React + TypeScript + Vite + Tailwind + Zustand + localStorage - categories are the main visual unit - progress is based only on active tasks - not_needed is reversible and excluded from progress/points - business logic must stay out of large UI components  If the requested change conflicts with the MVP scope, implement the smallest version that respects the existing architecture and scope.  ## Step 2 — Identify the layer being changed  Before editing files, classify the work into one or more of these layers:  - `catalog` - `rules` - `logic` - `store` - `components` - `types` - `utils`  Prefer touching the smallest number of layers needed.  Do not solve logic problems inside presentation components if they belong in `logic/`, `rules/`, or `store/`.  ## Step 3 — Implement in the simplest valid way  When writing code: - keep logic deterministic - use pure helper functions where possible - avoid hidden coupling - avoid speculative abstractions - avoid adding libraries unless clearly necessary - preserve readability over cleverness  Prefer: - small composable functions - clear derived selectors/helpers - explicit naming - low-magic state transitions  ## Step 4 — Protect critical product rules  Pay special attention to these rules:  ### not_needed A task marked as `not_needed`: - is not deleted - is excluded from active tasks - is excluded from progress - is excluded from points - remains recoverable later  ### progress - category status is derived, not manually stored - overall progress recognizes partial internal task completion - Trip readiness uses essential active tasks - Comfort extras uses extra active tasks  ### context editing - editing context regenerates the checklist in V1 - do not implement advanced state reconciliation  ## Step 5 — Keep the UI aligned with product intent  The UI should remain: - calm - clear - product-oriented - believable as a real MVP  Avoid: - noisy controls - childish game-like styling - cluttered task cards - visual complexity not justified by the MVP  If unsure, simplify.  ## Step 6 — Validate before finishing  Before concluding: - run available validation commands if present - check for obvious TypeScript errors - make sure imports/exports are consistent - remove accidental dead code - confirm the change still respects MVP scope  If commands are not available, say so explicitly.  ## Step 7 — Return a structured delivery summary  Always end with this structure:  ### Delivery summary - What changed - Files created or modified - Main technical decisions - Simplifications made - Risks or rough edges - What should be reviewed manually next  ## Additional guidance  When a request is ambiguous: - choose the simplest implementation that respects `AGENTS.md` - do not silently expand scope - do not invent "future-proof" abstractions unless they clearly reduce complexity now  When refactoring: - preserve behavior unless the request explicitly changes it - keep folder responsibilities clean - prefer improving clarity over reorganizing everything  When implementing a new feature: - build the smallest coherent slice - keep it easy to review - leave obvious extension points only where they are genuinely useful
